# GBrain CLI Lock Wedge — Cron Overlap

## Symptom (Vince 2026-06-26 15:58 SGT, #general)
"Fix Gbrain issue" — `gbrain capture` and `gbrain doctor` were timing out / SIGKILLed. `gbrain status --fast` also wedged. Supervisor appeared online but unresponsive.

## Root Cause (5-layer failure cascade)

1. **Autopilot cycle takes ~13 min** (per cron log: ran all phases, then idle-poll for jobs)
2. **Cron fires every 30 min** (`*/30 * * * *`), but the autopilot worker **does NOT exit after cycle completes** — it keeps polling for new jobs in `ep_poll`
3. **Wrapper script holds lock indefinitely** while child bun worker is alive (cleanup trap only fires when bash exits, which requires worker to exit)
4. **No `flock -n` at the cron level** — overlapping cron runs both end up waiting 60 min for the same lock (LOCK_TIMEOUT=3600s)
5. **v0.42.52.0 supervisor self-heal didn't catch this** — the wedge was at the cron wrapper level, not the supervisor's worker

## Cascade diagram (before fix)
```
cron 16:00 → wrapper 30633 → autopilot 42878 → jobs work wrapper 43156 → worker 54760
cron 16:30 → wrapper 4084050 (waits 60 min for lock; or 13 min in practice after PID 43156 died)
cron 16:30+ doctor requests → all blocked on .gbrain-cli.lock
```

## Detection (SIGKILL pattern)
- `gbrain doctor --fast` exit 124 (timeout) after 60s
- `gbrain capture` exit code 0 but no output (silent hang)
- Lock file `/root/.gbrain/.gbrain-cli.lock/lock` always present with `{pid, acquired_at, command}` JSON
- Multiple overlapping `gbrain jobs work` processes visible in `ps aux`

## Fix applied (2026-06-26 16:55 SGT)

### 1. Killed stuck autopilot groups
- Group 1 (cron run 16:00): PIDs 30616, 30633, 42878, 157281, 157684 → SIGTERM (clean exit, lock released via trap)
- Group 2 (cron run 16:30): PIDs 4084037, 4084050, 157689, 157763, 174072 → SIGTERM (clean exit)

### 2. Added `flock -n` to cron entry
```diff
-*/30 * * * * /root/scripts/gbrain-autopilot-cron.sh >> /root/.openclaw/workspace/logs/gbrain-autopilot-cron.log 2>&1
+*/30 * * * * /usr/bin/flock -n /var/lock/gbrain-autopilot.lock /root/scripts/gbrain-autopilot-cron.sh >> /root/.openclaw/workspace/logs/gbrain-autopilot-cron.log 2>&1
```
`flock -n` is **non-blocking** — if the lock is held, the new cron invocation exits immediately instead of waiting. Tested with real gbrain command: second invocation rejected with exit 1.

### 3. PM2 supervisor restart
`pm2 restart gbrain-jobs-supervisor` — restarts from 1 → 2. Process now writes to fresh log, health check loop active again.

### 4. Reinstalled autopilot
`gbrain autopilot --install` — installs systemd user service `gbrain-autopilot.service` for the new checkout path.

## Verification
- `gbrain doctor --fast`: returns in <5s (was: 60s timeout, then SIGKILL)
- `gbrain status --fast`: returns in <5s with full output
- `gbrain capture --help`: returns immediately
- Lock files: cleared
- gbrain processes: only `gbrain.real serve --http --port 3050` (MCP) — clean
- `pm2 list`: gbrain-jobs-supervisor online, 95.9mb, 2 restarts
- Doctor health score: 60/100 (resolver_health fail = pre-existing skill registration issues, NOT related to wedge)

## Lessons

1. **Lock cleanup must be tied to operation completion, not process lifetime.** The bash wrapper's trap only fires on bash exit. Worker doesn't exit after cycle completes → lock held indefinitely.

2. **`flock -n` at the cron level is essential** for any daemon that might run longer than its cron interval. The internal lock mechanism alone is insufficient — it serializes within the daemon but doesn't prevent daemon multiplicity.

3. **The v0.42.52.0 supervisor self-heal** addresses worker crashes within a supervisor-owned worker. But cron-spawned wrappers are NOT supervised — they have their own lifecycle independent of PM2.

4. **Pre-existing diagnostic noise** (`home_dir_in_worktree`, `resolver_health` skill failures) made it harder to spot the real wedge. Not blocking — flagged for separate cleanup.
